Constitution Rules

Whereas Profiles describe Payment Orders and Remittances in terms of banking standards and formats, Constitution Rules specify the way remittances are generated, depending on the provided banking services.

Example

most of your banks work with the MT101 format. This format provides transfers that are multi-currency, multi-account, etc. You have consequently created a common profile for these banks in the module.

However, all the banks do not grant the totality of the options provided by the MT101 format; as a result, your conditions differ from one bank to another. You need to create as many Constitution Rules as specificities.
You need to create at least one Constitution Rule by transaction type.

Remittances can be:

  • Multi-currency: They contain orders, whatever the currencies.
  • Single-currency: They contain orders expressed in the same currency.
  • Multi-date: Their orders can have different execution dates.
  • Single-date: They contain orders planned for a same execution date.
  • Limited to a maximum number.
  • Global (global debit of the remittance or by currency) or detailed by transaction (at the level of the account statement).

As Constitution Rules are linked to the set up Profiles, some options provided by these rules can be limited if the format requires it. For instance, if you create a multi-date constitution rule for a given branch, and that some orders selected for the remittance depend on the MT101 format (which is single-date), then these orders are remitted by date.

Likewise, the AFB320 format requires that the orders are remitted by applicant, which means that all orders belonging to this format, will be remitted by applicant.

Like the Profiles setup, the setup for Constitution Rules is only made once by the Administrator. The names of these rules give enough information for a basic usage. In practice, when Profiles and Constitution Rules have been defined, the user with the rights to enter and/or generate remittances has very little left to do.
